# 테스트 주도 개발 (TDD) 원칙

본 프로젝트는 비즈니스 로직의 안정성과 무결성을 확보하기 위해 TDD(Test-Driven Development) 사이클을 원칙으로 합니다.

## 1. 테스트 작성 우선 (Test First)
- **UI를 제외한 코어 로직은 실패하는 테스트를 먼저 작성합니다.**
- 테스트 없이 새로운 코어 기능을 구현하지 않습니다.
- 테스트는 요구사항과 명세(Spec)의 역할을 대체합니다.

## 2. TDD 사이클 (Red-Green-Refactor)
개발은 반드시 다음 세 단계를 반복하여 진행합니다.
1. **Red**: 실제 동작하는 구현 코드를 작성하기 전에, 실패하는(컴파일 에러를 포함한) 테스트 코드를 가장 먼저 작성합니다.
2. **Green**: 실패하는 테스트를 통과시키기 위한 가장 단순하고 최소한의 코드를 작성합니다.
3. **Refactor**: 테스트를 통과시킨 후, 중복을 제거하고 코드의 구조(가독성, 성능)를 개선합니다. 이때 테스트 코드가 시스템의 정상 동작을 보장합니다.

## 3. 기능 단위 분리
- **한 번에 하나의 동작만 테스트합니다.**
- 테스트 메서드의 범위가 너무 커지면 설계를 의심하고 기능을 더 작게 분리합니다.
- Given-When-Then(또는 Arrange-Act-Assert) 구조를 사용하여 테스트의 의도를 명확하게 드러냅니다.
