# 객체 지향 및 SOLID 설계 원칙

유연하고 확장 가능한 시스템 설계를 위해 SOLID 원칙을 기반으로 합니다. 프로젝트의 구조적 일관성을 유지하는 가장 중요한 지침입니다.

## 1. 단일 책임 원칙 (SRP)
- **단일 책임 원칙을 따릅니다.**
- 모든 클래스, 모듈, 함수는 단 하나의 변경 이유만 가져야 합니다.
- 데이터 모델 검증, 외부 API 호출, 프롬프트 조합 작업은 각각 별도의 클래스(또는 파일)로 분리합니다.

## 2. 인터페이스 분리 (ISP / DIP)
- **AI Provider, 저장소(Repository), 분석 서비스는 인터페이스로 분리합니다.**
- 외부 시스템(OpenAI API 연동 모듈, SQLite/PostgreSQL 데이터베이스)과 내부 비즈니스 로직의 직접적인 결합을 피합니다.
- 특정 LLM 업체(예: OpenAI)에 코어 로직이 직접 의존하지 않도록 의존성 역전(Dependency Inversion) 원칙을 적용합니다.

## 3. 도메인과 관심사 분리
- **UI 코드와 도메인 로직을 철저하게 분리합니다.**
- 프론트엔드 컴포넌트는 사용자와의 시각적 상호작용 및 API 호출에만 책임집니다.
- 분석을 처리하는 서버 레이어는 HTTP 요청 처리(Router)와 비즈니스 수행(Service) 레이어를 격리합니다.
